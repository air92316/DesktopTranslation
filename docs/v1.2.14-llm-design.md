## 1. AppSettings.cs schema 變更

### 新增/變更欄位

| 欄位 | 型別 | 預設值 | 來源/原因 |
|---|---|---|---|
| `LlmProvider` | `string` | `"claude"` | **保留**（原欄位） |
| `ApiKey` | `string` | `""` | **保留**（向後相容，當作 active provider 的 key） |
| `LlmModel` | `string` | `""`（空字串=用 provider 預設） | **新增**：使用者自選或自訂 model id |
| `LlmBaseUrl` | `string` | `""`（空字串=用 SDK 預設 endpoint） | **新增**：OpenAI 相容服務轉址；僅對 OpenAI provider 生效 |
| `LlmTemperature` | `double` | `0.3` | **新增**：翻譯任務適合低 temperature；0.0–2.0 範圍 |
| `LlmMaxTokens` | `int` | `2048` | **新增**：上限保護成本；輸入上限 5000 chars 翻譯通常 <2048 token |
| `OpenAiApiKey` | `string` | `""` | **新增（per-provider key）** |
| `ClaudeApiKey` | `string` | `""` | **新增（per-provider key）** |
| `GeminiApiKey` | `string` | `""` | **新增（per-provider key）** |

### 為什麼要分 per-provider key？

**現況痛點**：使用者切換 provider 必須重新貼 API key。三家服務的 key 互不通用，使用者很可能同時擁有 OpenAI + Claude 的 key 想做 A/B 比較。

**設計**：保留 `ApiKey` 欄位（**向後相容 v1.2.13 已存的設定**），同時新增三個 per-provider 欄位。讀取邏輯：
```
effectiveKey = LlmProvider 對應的 per-provider key (若有值)
              else fallback 到 ApiKey (舊版設定)
```

寫入邏輯：使用者輸入 key 時，**同時寫入** per-provider 欄位 **和** `ApiKey`（讓舊欄位反映 active provider 的 key，保持單一真實來源往前看）。

> **遷移路徑**：v1.2.13 → v1.2.14 第一次啟動，把舊 `ApiKey` 內容自動複製到當前 `LlmProvider` 對應的 per-provider 欄位即可，使用者完全無感。

### LlmModel 預設值的設計選擇

兩種方案：
- **A：欄位預設 `""`，由 engine 在執行時 fallback 到 provider 預設常數**（推薦）
  - 優點：未來 provider 升級預設 model 時，不需要 migration；舊使用者自動受益
  - 缺點：UI 需要分辨「未自訂」與「自訂為某固定值」
- **B：欄位預設寫具體 model id（例如 `"gpt-5.4-nano"`）**
  - 優點：UI 邏輯單純
  - 缺點：使用者升級到 v1.2.15 時若我們想改預設，舊使用者卡在舊 model

**選 A**。UI 上「Model 下拉選單」第一項顯示「使用預設（gpt-5.4-nano）」，括號內動態顯示當前 provider 預設值。

### Schema 安全性

`ApiKey` 系列欄位目前以明碼存在 settings.json（v1.2.13 既有行為）。v1.2.14 **暫不改變**這個行為，避免擴大 scope。但需在文件留筆：未來考慮以 DPAPI 加密（已引入 `System.Security.Cryptography.ProtectedData` NuGet，但尚未使用）。

## 2. Provider 抽象設計

### 2.1 是否需要 Provider enum？

**建議：新增 `LlmProviderKind` enum**，但**內部用，不改 settings 欄位型別**。

```
理由：
- AppSettings 用 string 持久化（向後相容、易讀）
- 內部分支邏輯用 enum（編譯期保證、避免 typo）
- enum ↔ string 的轉換集中在 Parse 函式
```

enum 候選成員：`Claude`、`OpenAi`、`Gemini`。

### 2.2 LlmTranslateEngine 內部結構

**現況**：單一檔案 169 行，內部 if/else 切 provider。新增 Gemini 後若繼續用 if/else 會超過 250 行，且每個 provider 的錯誤分類規則不同。

**推薦：輕量 Strategy pattern，每個 provider 一個內部類**。

```
LlmTranslateEngine (主類，public)
├── 建構子接收 AppSettings 子集（provider/model/baseUrl/key/temperature/maxTokens）
├── 依 provider 在建構時建立對應的 IProviderClient 實例
└── TranslateAsync: 委派給 provider client，再做共用的 prompt wrap、source detect、error classify

internal interface IProviderClient
├── Task<string> CompleteAsync(systemPrompt, userText, ct)
└── ErrorKind ClassifyError(Exception ex, CancellationToken ct)

internal class ClaudeProviderClient : IProviderClient
internal class OpenAiProviderClient : IProviderClient   (含 BaseUrl 支援)
internal class GeminiProviderClient : IProviderClient
```

**為什麼不用完整 DI/factory pattern**：
- 專案規模小（13 個 service，無 DI container）
- 每個 provider client 都很薄（~30 行）
- 建構時依 provider enum switch new 一個就好，不需要註冊機制

**檔案組織建議**：
- `Services/Llm/LlmTranslateEngine.cs`（主類）
- `Services/Llm/IProviderClient.cs`
- `Services/Llm/ClaudeProviderClient.cs`
- `Services/Llm/OpenAiProviderClient.cs`
- `Services/Llm/GeminiProviderClient.cs`
- `Services/Llm/LlmModelCatalog.cs`（model 預設清單，靜態資料）

> 走 `Services/Llm/` 子資料夾是因為 v1.2.14 後 LLM 相關檔案會從 1 個變成 6 個，平鋪會吃掉 `Services/` 目錄的可讀性。

### 2.3 共用邏輯抽出

下列現在散在 `LlmTranslateEngine` 的邏輯應留在主類（不下沉到 provider client）：
- Prompt 組裝（system prompt + `<translate>` XML wrap）
- 5000 char 截斷
- `LanguageDetector.DetectSourceLanguage` 呼叫
- `TranslationResult` 組裝
- 共用的 `HasStatusCode` / `ContainsSocketException` 工具

provider client 只負責「文字進、文字出，丟原生 SDK 例外」+ 自家錯誤分類。

### 2.4 Gemini SDK 選擇

候選方案：

| 方案 | 套件 | 優點 | 缺點 |
|---|---|---|---|
| **A. Mscc.GenerativeAI** | `Mscc.GenerativeAI 3.1.0`（社群） | 功能全、文件齊、追新版 model 很快 | 非 Google 官方；單一作者維護有 bus factor 風險 |
| **B. Google_GenerativeAI** | `Google_GenerativeAI 3.6.6`（社群） | 也是社群活躍 | 同上 bus factor；命名跟官方混淆 |
| **C. 直接 HttpClient** | 無新增套件 | 0 新依賴；完全可控；endpoint 簡單 | 需自行處理 JSON schema、錯誤碼、未來 API 變化 |

**推薦：方案 C（直接 HttpClient + System.Text.Json）**。

理由：
1. **本專案的 Gemini 使用面極窄**：只用 generateContent，不用 streaming、function calling、grounding、vision。SDK 提供的 90% 功能用不到。
2. **減少 NuGet 攻擊面**：v1.2.13 已有 7 個 PackageReference，加 Mscc 會綁住一個第三方維護者的更新節奏。Anthropic SDK（Claudia）也是社群維護的選擇，但那個沒有官方替代；Gemini 的 REST API 極穩定，自寫成本低。
3. **endpoint 真的很簡單**：
   ```
   POST https://generativelanguage.googleapis.com/v1beta/models/{model}:generateContent?key={apiKey}
   Body: { "contents": [{ "parts": [{ "text": "..." }] }], "systemInstruction": {...}, "generationConfig": {"temperature": 0.3, "maxOutputTokens": 2048} }
   ```
4. **Claudia 是社群也跑得好好的，Gemini 也用同樣標準**：Claudia (1.3.2) 與本專案的 OpenAI 官方 SDK 並存兩年沒問題，Gemini 用 HttpClient 會更輕。

**Trade-off 揭露（規則 18.1）**：HttpClient 寫法未來若要加 streaming 或 function calling，要自己 parse SSE，比 SDK 麻煩。但本專案 v1.2.14 範圍不需要這些，且未來真的需要時再切回 SDK 不會破壞 IProviderClient 介面。

## 5. 連線測試 flow

### 5.1 測試文字選擇

**推薦：「你好」→ 英文**

理由：
- 短（節省成本，4 token 上下）
- 跨語言（驗證雙向偵測 + 翻譯，比同語言 echo 嚴謹）
- 中文對所有三家 provider 都熟悉，回 "Hello" 之類的失敗率低
- 不會踩任何 content moderation（不像「test」可能被 truncate 規則影響）

備註：UI 不暴露這個測試文字給使用者（避免顯示「我們在送你好」造成困惑）。

### 5.2 UX flow

```
Idle → 點擊 → Loading → Success | Failure
                          ↓
                          (可重試)
```

| 狀態 | UI 表現 |
|---|---|
| Idle | 按鈕顯示「連線測試」，TxtTestStatus 為空 |
| Loading | 按鈕 IsEnabled=false 顯示「測試中…」+ ProgressRing 或 dots 動畫；TxtTestStatus 顯示灰色「正在連線到 {provider}…」 |
| Success | 按鈕還原；TxtTestStatus 顯示綠色 ✓ 與「成功（{耗時 ms} ms）」 |
| Failure | 按鈕還原；TxtTestStatus 顯示紅色 ✗ 與分類錯誤訊息（見 5.3） |

**timeout**：6 秒（短於正常翻譯的 10 秒，避免使用者等太久）。

### 5.3 失敗錯誤分類（複用既有 ErrorKind）

| ErrorKind | 顯示訊息 |
|---|---|
| `ApiKey` | ✗ API Key 無效或未授權，請確認 key 是否正確 |
| `RateLimit` | ✗ 觸發速率限制，請稍候再試 |
| `Timeout` | ✗ 連線逾時（>6s），請檢查網路或 BaseUrl |
| `Network` | ✗ 網路連線失敗，請確認可連線到 {provider domain} |
| `Unknown` | ✗ 連線失敗：{exception.Message}（截斷到 100 字） |

> **不揭露 stack trace 或完整 exception**（規則 5：不洩漏內部細節）。技術細節寫進 `Debug.WriteLine`。

### 5.4 與正式翻譯路徑共用程式碼

連線測試 **不繞過** `LlmTranslateEngine` 路徑，直接呼叫 `engine.TranslateAsync("你好", "en", ct)`。

**為什麼**：保證測試成功 ⇒ 正式使用會成功。如果繞過 prompt wrap、source detect 等共用邏輯，可能測試 OK 但實際翻譯壞掉，違反「測試 = 真實使用」的原則。

唯一差別：建立一次性的 engine 實例（用 UI 上「目前未存檔」的 settings），不影響已運行的 TranslationService 主流程。

## 6. 風險與 trade-off（self-critique）

### 風險 1：BaseUrl 自訂的安全面

**問題**：使用者填入惡意 endpoint（例如釣 API key 的 phishing proxy），程式會把真實的 OpenAI API key 直接送過去。

**緩解**：
- UI 在 BaseUrl 欄位下方加警示文字：「⚠️ 此 URL 將收到您的 API key，請只填您信任的服務。」
- 文件（README）列出常見可信替代 endpoint（OpenRouter、Groq、Together）的官方網址做白名單參考
- **不主動白名單檢查**：黑/白名單策略會擋掉合法用例（自架 LiteLLM proxy、Ollama），且容易過時

**接受程度**：使用者自訂 BaseUrl 本質上就是進階功能，承擔風險是合理的；風險訊息透明就好。

### 風險 2：Gemini 加進來增加維護面

**問題**：v1.2.14 後維護面從 2 provider 變 3 provider，未來 model 更新、API 變動的工作量增加 1.5 倍。

**緩解**：
- IProviderClient 抽象讓每個 provider 邊界清楚，加新 provider 不影響其他兩家
- 用 HttpClient 而非 SDK 減少第三方依賴鎖定
- LlmModelCatalog 集中管理 model id 清單，每 3 個月（或主要 provider 發新 model 時）跑一次 audit

**接受程度**：使用者明確要求要 Gemini，這個成本是必要的。

### 風險 3：Model id 寫在 settings 裡，未來 model 退役怎麼辦

**問題**：v1.2.14 預設 `gemini-2.5-flash`，假設 2027 年該 model 退役，使用者升級到 v1.2.20 後，已存的 settings.json 還寫著舊 model id，會直接打 API 拿到 404。

**緩解**：
- 預設值用空字串 + runtime fallback 到 `LlmModelCatalog.GetDefault(provider)`（見 1. schema 設計）
- 如果使用者自選了某個 model 然後該 model 退役，`LlmTranslateEngine` 收到 API 404 時應該回傳 `ErrorKind.Unknown` 並把錯誤訊息設為「Model {id} 已停用，請到設定改選」
- 加 settings migration：偵測到 settings 裡的 model id 不在當前 catalog 且**也不像自訂模式**時，提示使用者重設

**接受程度**：這是所有「外部依賴版本綁定」的固有風險，無法消除只能緩解。

### 風險 4（self-critique 補充）：per-provider key 同步邏輯易錯

**問題**：第 1 節設計同時寫入 `ApiKey` 和 `XxxApiKey`，若邏輯不對稱可能出現「使用者改了 OpenAI key，Claude key 不知為何也變了」。

**緩解**：明定**單向同步**規則：
- 寫入：使用者輸入 → 寫到 `{provider}ApiKey` + 同步到 `ApiKey`
- 讀取：從 `{provider}ApiKey` 讀；空才 fallback 到 `ApiKey`
- **絕不雙向同步**

需要在 SettingsService 寫 unit test 覆蓋 v1.2.13 → v1.2.14 升級場景。

### 風險 5（self-critique 補充）：連線測試會吃使用者 quota

**問題**：每次點按鈕都實際呼叫 API，使用者可能不知道「點測試 = 花錢」。

**緩解**：按鈕 tooltip 註明「將實際呼叫 API（約 $0.0001 USD）」。費用其實微乎其微但透明告知。

### 能否並行？

**Phase 2a 與 2b 不能並行做**：

- 並行會兩邊都改 SettingsWindow.xaml 造成 merge 地獄

**Phase 2a 內部可拆 sub-batch**：
- Sub-A：schema + catalog + engine refactor（只改 backend）
- Sub-B：SettingsWindow UI（依賴 Sub-A）
- Sub-A 與 Sub-B 不能完全並行（UI 引用 catalog），但 Sub-A 完成後 Sub-B 就獨立。

**Phase 2b 內部可拆**：
- Sub-C：Gemini provider client（純 backend）
- Sub-D：連線測試 UI（純 UI；不依賴 Gemini，可在 Phase 2a 後就開始）

→ 若想加速：Phase 2a 結束後 Sub-C + Sub-D 可並行（不同檔案、無共享 state）。
